## 背景与目标
- 目标：把“会话分析”的关键生命周期事件（MCP StartTasks、core 运行任务）统一上报到服务端，由服务端通过 CLI 包内现有的 vf-call-hook 触发插件 hook，让插件自行沉淀分析数据/指标。
- 约束：不新增 CLI subcommand；复用 apps/cli 的 vf-call-hook；事件名使用 PascalCase；不引入 Ready/Fail 事件，失败通过 TaskStop 的状态表达。

## 现状复用点（已存在）
- vf-call-hook 实际执行入口：apps/cli/call-hook.js → apps/cli/src/hooks/index.ts（从 stdin 读 JSON 并触发插件 hook）。
- Hook 输入结构：packages/core/src/hooks/type.ts 里的 HookInputCore/HookInputs（当前只有 PreToolUse/SessionStart 等 Anthropic 风格事件）。
- MCP StartTasks 真实触发点：apps/cli/src/mcp-tools/task/index.ts 的 StartTasks handler。
- core 运行 adapter 的入口：packages/core/src/controllers/task/run.ts 的 run()。

## 事件模型（PascalCase）
- StartTasks：表示 MCP 工具 StartTasks 被调用（一次调用可能包含多个 task）。
- TaskStart：表示某个 task 的 adapter.query 即将开始执行（对应一个 task/session）。
- TaskStop：表示 task 结束（成功/失败）——失败不单独发事件，通过 stopStatus 标识。

## Hook 输入扩展（核心类型改造）
- 在 packages/core/src/hooks/type.ts 的 HookInputs 增加 3 个新 key：StartTasks / TaskStart / TaskStop。
- 每个事件都沿用 HookInputCore（cwd/sessionId/hookEventName/transcriptPath）+ 事件专属字段：
  - StartTasks：{ parentSessionId?: string; tasks: Array<{ taskId: string; name?: string; type?: string; agents?: number[] }>; source: 'mcp' }
  - TaskStart：{ taskId?: string; parentSessionId?: string; adapter?: string; model?: string; source: 'core' | 'mcp' }
  - TaskStop：{ taskId?: string; stopStatus: 'Completed' | 'Failed' | 'Cancelled'; exitCode?: number; errorMessage?: string; source: 'core' | 'mcp' }
- transcriptPath：由于这不是 Anthropic transcript 语境，服务端统一填空字符串或固定占位（例如 ''），插件按需忽略。

## 服务端接口（统一入口）
- 新增接口（示例）：POST /api/hooks/call
- 入参：{ hookEventName: 'StartTasks' | 'TaskStart' | 'TaskStop'; input: Record<string, unknown>; env?: Record<string, string> }
  - input 会被服务端补齐为完整 HookInput（至少包含 cwd/sessionId/hookEventName/transcriptPath）。
  - env 允许透传 __VF_PROJECT_WORKSPACE_FOLDER__、__VF_PROJECT_AI_CTX_ID__、__VF_PROJECT_AI_LOG_PREFIX__ 等（可选）。
- 行为：
  - 仅允许白名单 hookEventName（防止任意执行）。
  - 使用 child_process spawn 调用“node apps/cli/call-hook.js”（或解析到 vf-call-hook bin 文件），把 HookInput JSON 写入 stdin。
  - 收集 stdout JSON（CLI hooks 会输出插件执行结果 JSON），作为接口响应返回。
  - 失败策略：接口返回错误信息，但调用方（MCP/core）应以“失败不阻塞主流程”为原则（上报失败不影响任务执行）。
- 性能/稳定性：服务端对该接口内部增加并发限制（例如同时最多 N 个 hook 子进程），避免频繁事件导致进程风暴。

## 触发点接入
### 1) MCP StartTasks → 调用服务端接口
- 在 apps/cli/src/mcp-tools/task/index.ts 的 StartTasks handler：
  - resolvedTasks 生成后立即调用 /api/hooks/call，hookEventName=StartTasks。
  - sessionId 选择：优先使用 parentSessionId（即 __VF_PROJECT_AI_CTX_ID__），保证日志与会话上下文对齐。
  - payload 包含 taskId 列表、任务描述、agents 等。

### 2) core 运行 adapter → 调用服务端接口
- 在 packages/core/src/controllers/task/run.ts 的 run()：
  - adapter.query 之前上报 TaskStart（携带 sessionId=adapterOptions.sessionId、adapter/model/taskId 等）。
  - wrappedOnEvent 捕获 exit 时上报 TaskStop：
    - stopStatus：exitCode===0 → Completed，否则 Failed。
    - exitCode：透传。
    - errorMessage：可从异常/exit 事件补充（若有）。
- 服务端地址获取：复用 CLI 现有约定（__VF_PROJECT_AI_SERVER_HOST__/__VF_PROJECT_AI_SERVER_PORT__），core 在 Node 环境下可直接读取 process.env 拼 baseUrl。

## 安全与隔离
- /api/hooks/call 默认仅允许本机访问或需要简单 key（与现有 automation webhook 类似的 key 模式即可），避免远程任意触发本地进程。
- 透传到 vf-call-hook 的 env 严格白名单，禁止透传敏感或可注入的变量。

## 验证方式
- 单元验证：
  - 直接调用 /api/hooks/call 触发 StartTasks/TaskStart/TaskStop，验证 vf-call-hook 能跑通并返回插件输出。
- 端到端验证：
  - 从 MCP StartTasks 发起，确认 server 收到 StartTasks 并成功触发 hook。
  - 启动一个 task，确认 core 在 TaskStart/TaskStop 都触发且 stopStatus/exitCode 正确。
